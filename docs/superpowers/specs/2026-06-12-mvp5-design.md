# MVP-5 design — 场景实证 + 通用工具面板 + 主题自定义层

Status: owner ratified scope（2026-06-12 对话）；详细决策见
[mvp5-scope-2026-06-12.md](../../engineering/design/discussions/mvp5-scope-2026-06-12.md)。

## Goals

1. **Phase 0 场景实证**：三个真实使用场景案例（日记 / 旅行照片 / 代码解析）落进 owner dev 库，并以 v2 导出包形式进仓库 `examples/`，成为产品 demo + 主题测试夹具 + 一键导入示例包。
2. **Phase 1 通用工具面板**：模块声明工具、host 渲染的贡献点（M5-D4 分权规则）+ 主题化表单原语（M5-D5）。这是 plugin-system 主线前的 chrome 障碍清除。
3. **Phase 2 主题自定义层**：主题授权的调色板变体 + token 覆写白名单（M5-D3，自由调色 rejected）；自定义进导出格式（v3，成对 transform）。

## Non-goals

- Modal/弹层体系、通用文件选择器、插件设置页（等 plugin-system 正轮）
- 逐页自定义（钉选只钉 theme id；自定义为实例级、按 themeId 分键存储）
- 对比度/可达性自动校验（变体由主题作者策展——这正是该模型的意义）
- 动态插件注册（registry 仍静态；本轮只把贡献点的形状做对）

## Phase 0 — 场景案例

**形态**：可重放的 seed 脚本 `scripts/seed-examples.ts`（走 HTTP API，admin 凭据 + base URL 参数化——诚实路径，压测真实产品面）。同一脚本跑两次：

1. 对 owner dev 库（:3210）——内容进他的工作区（owner 拍板直接建在 dev 库），独立文件夹 `示例`
2. 对 scratch 实例（独立 SHCKB_DB_PATH 临时库）——干净导出 → `examples/*.zip`（owner 真实页面不进仓库）

**三案例**（每个 = 1 文件夹 + 1-2 页，全部 publish）：

| 案例 | 钉选主题 | 压测点 |
|---|---|---|
| a. 日记《六月手帐》 | stationery | markdown 长文 + 手写体 + 撕边/胶带在真实文字密度下的观感 |
| b. 旅行照片《海岸线三日》 | stationery | 拍立得多图排版、image+markdown 混排、blob 管线 |
| c. 代码解析《grid-engine 重力算法讲解》 | workbench | code(draft) 多块 + markdown 交替的讲解流、高亮密度 |

图片：本地脚本生成确定性抽象 PNG（渐变/几何，无版权问题；SVG 被 blob 管线排除故必须位图）。

**产出物**：`examples/README.md`（每包一行说明 + 导入指令）+ 3 个 zip。摩擦清单记入 build log——它们是 Phase 1/2 的需求输入（M5-D1）。

## Phase 1 — 表单原语 + 工具面板

### packages/ui-kit（新 workspace 包 @skb/ui-kit）

依赖方向：`ui-kit → theme`；`block-kinds → ui-kit`；`web → ui-kit`。⚠️ Dockerfile 需补 COPY（mvp-4 实测坑）。

原语（全部 token 驱动，消费 useTheme）：

```ts
UiSelect:    { value, onChange, options: {value,label}[], title? }
UiButton:    { onClick, children, variant?: 'default'|'accent'|'danger', title? }
UiTextInput: { value, onChange, placeholder?, mono? }
UiToggle:    { checked, onChange, label }
```

样式来源：accent / hairline / surfaceInsetBg / textColor / mutedColor / blockRadius。已知消费者：code 语言选择（迁移）、image alt 工具（新）、Phase 2 自定义 UI（调色板/字体/颜色）、Sidebar 既有 select 逐步换装（本轮只换 theme 相关两处，避免无界重构）。

### 工具面板贡献点

```ts
// types.ts 扩展
export type BlockToolProps<C = unknown> = { content: C; onChange: (next: C) => void };
export type BlockKindModule<C = unknown> = {
  ...现有字段;
  /** 激活块的工具，host 在自己的面板里渲染（插件管内容，host 管位置/布局）。
   *  工具组件用 ui-kit 原语拼装；可经 useHost() 拿 host 能力。 */
  tools?: Array<{ id: string; label: string; View: ComponentType<BlockToolProps<C>> }>;
};
```

Host 侧（apps/web 新 `grid/ToolPanel.tsx`）：固定面板（Palette 下方，同视觉语言），仅激活块存在且其模块声明了 tools 时显示；渲染 `[label: View]` 列表，传当前 content + onContentChange。无激活块/无工具 → 不渲染。form factor 是 dev 决策，不是产品锁。

**迁移证明**（贡献点的真实消费者）：
- code kind：语言选择从 EditView 内嵌迁出为 tool（EditView 只剩 textarea）
- image kind：新增 alt 文本 tool（RenderView 已有 alt 消费路径则接通；无则补）

PublishedCanvas / 静态渲染不受影响（tools 只在编辑面存在——RenderView 纯净规则不变）。

### Palette 正式化

已是 registry 驱动；本轮：CONTRACT.md 增"贡献点"章（palette 条目 = kind/label/glyph 契约 + tools 契约 + 分权规则），代码零硬编码复核。不加分类/搜索（YAGNI，三个 kind 用不上）。

## Phase 2 — 主题自定义层

### 类型（packages/theme）

```ts
export type PaletteVariant = { id: string; name: string; tokens: Partial<ThemeTokens> };
export type Theme = ThemeTokens & ThemeSlots & {
  /** 主题策展的官方配色变体（科研 colormap 式）；缺省 = 无变体可选 */
  palettes?: PaletteVariant[];
  /** 开放给实例 operator 直接覆写的 token 白名单（opt-in；缺省 = 全锁） */
  customizableTokens?: (keyof ThemeTokens)[];
};
export type ThemeCustomization = { paletteId?: string; overrides?: Partial<ThemeTokens> };
/** 纯函数：base + 变体 + 白名单过滤后的覆写 → 合成 Theme（slots 原样保留） */
export function applyCustomization(base: Theme, c: ThemeCustomization | undefined): Theme;
```

合成顺序：base tokens → palette variant tokens → overrides（仅白名单内的键生效，越权键静默丢弃并不入库——server 写入时即过滤）。

**首批策展**：workbench 2 变体（如 warm/cool）、stationery 2 变体（如 牛皮纸/薄荷）、blueprint 1 变体（如 sepia 旧图纸）；graph-paper / ink 开 `customizableTokens: ['fontFamily', 'accent']` 证明白名单路径。具体值实现时定，原则：变体只动颜色族 token，不动几何（slot/pad/blockRadius 不进变体）。

### 存储与解析（apps/server）

- settings 新键 `themeCustomization` = JSON `Record<themeId, ThemeCustomization>`（按主题分键——切主题不丢各自的自定义）
- `effectiveTheme()` 变为：解析 id（钉选 ?? 实例 ?? 默认）→ `applyCustomization(THEMES[id], customization[id])`。publishedHtml = f(doc, slug, effectiveTheme) 签名与纯度不变，自定义经 effectiveTheme 流入
- 端点：PUT `/api/admin/settings/theme-customization`（admin；写前白名单过滤 + 未知 themeId/paletteId 422）→ 成功后 `rerenderAllPublished()`（现成机制）；GET 同位
- web Shell 的 instance theme 解析同步消费自定义（chrome 跟随）

### UI（apps/web，admin）

Sidebar 主题区扩展：当前实例主题有 palettes → 变体滑块/色板行（ui-kit 新 `UiPaletteSwatches`：每变体一组色点 + 名称，选中描边 accent）；有 customizableTokens → 对应控件（fontFamily → UiSelect 预置字体栈列表；颜色 token → 原生 color input 包装）。改动即 PUT + confirm 全量重渲染提示（与换主题一致）。

### 导出格式 v3

- manifest.settings 增可选 `themeCustomization`；FORMAT_VERSION = 3
- **成对 transform**：up(v2→v3) 无损（补缺省空对象/缺字段容忍）；down(v3→v2) 丢 `themeCustomization` 并逐项报损失（"instance theme customization dropped"）——旧 v2 构建 import 行为不变（只读 settings.theme，409 拒 v3 原生包的护栏自动生效）
- importer v3：读入后同样走白名单过滤（防手编包越权），未知 themeId 容忍（主题可能被移除——与 effectiveTheme 降级规则一致）

## 测试

- ui-kit：渲染冒烟 + token 接线断言（vitest，沿 theme 包模式）
- block-kinds：tools 契约测试（code 模块声明语言工具；工具变更经 onChange 往返）；RenderView 无工具泄漏断言
- theme：applyCustomization 纯函数全分支（变体合成 / 白名单过滤 / 越权丢弃 / 未知 paletteId）
- server：customization 端点（白名单过滤、422 路径、重渲染计数）；effectiveTheme 合成；export v3 round-trip + down transform 损失清单；v2 包 up 导入
- 容器：Dockerfile COPY ui-kit 验证（构建即测）
- Playwright 冒烟：工具面板出现/迁移后语言选择可用；变体切换后画布变色

## 风险与既知坑

- 新 workspace 包三件套：Dockerfile COPY / vite 重启 / bun TDZ（ui-kit 不建 context，纯消费 useTheme，环不会形成）
- 自定义覆写进 publishedHtml → 任何 customization 写操作必须触发重渲染（漏触发 = 不可再现状态，mvp-4 D2 同款不变量）
- seed 脚本对 owner 库：只建 `示例` 文件夹内内容，不触碰既有页面；中文 JSON 走 UTF-8 字节（PS 坑）
- format v3 是第二次实战迁移——损失清单必须逐项可读（mvp-4 标准）
