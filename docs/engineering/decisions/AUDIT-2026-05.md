## Why this audit

Phase B 批量起草 16 ADRs（2026-05-13），后续 owner reviews / external reviews / friend reviews 暴露：

- **(a)** 部分 ADR 误把 product vision / package contract 当 decision 写（ADR-0001 v1 / ADR-0003 v1）
- **(b)** Cross-ADR 一致性差（schema / type / 函数签名 / 引用漂移；ADR-0002 ↔ ADR-0007 ↔ ADR-0014 ↔ ADR-0017 互相不对齐）
- **(c)** 部分 frozen DI LOCK 没承接到 ADR（§11.11 install bootstrap / §9 grid theme / sidecar carrier 等）
- **(d)** ADR 颗粒度不一（architecture induction vs contract surface 边界模糊；ADR-0003 v1 / ADR-0014 都有这个倾向）

本 audit 出三件事：

1. 每个 ADR 当前 audit status（REWORKED / OK / MINOR-FIX / REWORK / GAP / PENDING）
2. Remaining debt register（hard blockers / soft debts / gap markers）
3. ADR rework round 2 的工作 list（**PRD-informed**——4 个 Day-1 PRD 写作过程中触碰到的 ADR 优先 rework）

## Methodology

每个 ADR 按以下维度 scan：

| 维度 | 问题 |
|---|---|
| Scope boundary | 有没有声明本 ADR 不决策什么？out-of-scope 表 |
| Induction vs contract granularity | 颗粒度对得上 ADR 性质吗？（参考 `process/methods/adr-discipline.md` foundational-ADR vs detail-contract 区分；audience-based 判定） |
| Cross-ADR 引用一致 | schema / types / 函数签名 cross-ref 没漂移？ |
| Frozen DI 覆盖 | source DI doc 的 LOCK 项是否都承接到 ADR？ |
| Reframe 信号 | 是否含 TS interface 大段 / 函数签名表 / 算法伪代码 / 文件路径细节（可能误把 CONTRACT 当 ADR 写） |
| 与新学到的 framing 原则一致 | "lived implementation pressure 之前不抽象 lock" / "append-only 是 phase-bound" 等 |

## Audit status legend

| Status | 含义 | Action |
|---|---|---|
| **REWORKED** | 已走 review-gate（owner / external / friend）+ pass 通过 | 等 owner accept；无进一步动作 |
| **OK** | scan 后判断结构与 contract 都 sound | 等 owner accept |
| **MINOR-FIX** | 小问题（cross-ref drift / typo / 措辞），不动 decision 本体 | 一次 batch fix |
| **REWORK** | 颗粒度 / scope / 一致性问题，要 rework 一轮（不是 supersede，是 pass 2 内修） | PRD-informed 后做 |
| **GAP** | 当前 ADR 未承接某个 frozen DI LOCK / 隐含决策；可能需新 ADR 或本 ADR 扩展 | 视性质 |
| **PENDING** | 还没 audit | TBD |

## Audit-time methodology lessons（喂回 process/methods）

本次 audit 暴露的方法论改进：

- **"Context 热就 lock" 是反模式** —— 在 lived implementation pressure 之前抽象 lock 易自造约束（ADR-0019 sidecar carrier 当下没开就是这条的应用）。Phase D framing-control retro 样本。
- **Append-only 是 phase-bound** —— 绑 `accepted` 状态；proposed 阶段可改 body。Phase D operational-lessons 样本。
- **Induction vs contract 颗粒度判定** —— audience-based（外部扩展点的 surface ∈ ADR；内部包 surface ∈ CONTRACT）。已写入 adr-discipline.md。
- **Foundational ADR induction-chain pattern** —— ADR-0001 / 0002 / 0003 / 0004 应同形态。已写入 adr-discipline.md。
- **Reframe 信号**（TS interface 大段 / 函数签名表 / 算法伪代码 → 误把 CONTRACT 当 ADR 写）。已写入 adr-discipline.md。
