# MVP-8 范围讨论 — 通用面板轮（2026-06-13）

Status: in progress（owner 开盘裁定 2026-06-13）

## 背景

MVP-7 收尾后挂账候选为 richtext kind / 页间一等链接 / 产品内搜索 / chrome 槽位契约。
owner 开盘裁定：在这些之前先做**通用面板**——右键菜单、设置界面、独立弹出面板
（删除确认等不再借用浏览器原生告警）。即此前提过的"通用面板样式"。

先后顺序的工程理由（讨论中确认）：

1. richtext kind 自身是浮层大户（链接编辑、格式菜单、slash 菜单都要锚定弹层），
   先有通用面板，richtext 落在现成原语上，避免再手写一批一次性 UI。
2. 通用面板是 chrome 槽位契约的前置——先把真实面做出来再冻结契约，顺序才对。

现状清点（开盘时）：

- 原生 `window.confirm/alert/prompt` 共 11 处，全部在 Sidebar.tsx 与 ThemeStudio.tsx。
- 右键菜单全仓为零；唯一的"菜单"是 Sidebar 里手写的 hover MoveMenu。
- 页级设置已收进 instruments 抽屉（M7）；实例级设置（主题/Studio/export/import/blob GC）
  仍平铺在 rack 脚部 Back office。
- ui-kit 只有 4 个表单原语，无浮层原语。

## 设计桩（讨论中钉定）

**一套面板解剖 + 三种机制，不是三个组件。**
右键菜单、设置面板、确认对话框共享同一套 Paste-Up 面板语言
（纸面、hairline 边、small-caps 标签头、press 按钮、危险动作 registration red）；
差别只在机制：模态（焦点锁定+Esc）、锚定（popover）、光标锚定（context menu，键盘可导航）。

**层归属：面板系统是 chrome，不是 theme。**
全部是作者侧仪表（匿名读者零仪表原则——读者一个菜单都不该见到）。
住 `apps/web/src/chrome/`，与 bench.tsx 同家；**不进 ui-kit**（ui-kit 是 theme 声部）。
天然落入未来 L1 UI 包契约范围 [层占据模型，theme-system PRD]。

> "可以一起铺"

## 工作清单

1. 浮层原语（`apps/web/src/chrome/`）：Dialog（confirm/prompt/alert 替代，Promise API）、
   Popover、ContextMenu，共享面板解剖。
2. 11 处原生调用全部迁移；MoveMenu 收编进 Popover。
3. 右键菜单：rack 行（重命名/移动/删除/此处新建）+ 画布（块：删除等；空白：插入块）。
   hover 小图标随之精简（动作不丢，行变干净——"动作重、设置敛"同向）。
4. 设置面板：Back office 四件套收进模态面板，rack 脚部只留一个入口。
