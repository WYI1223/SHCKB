## Theme

**From local toy to deployable, shareable instance.** MVP-1 proved the author loop locally; the product's core promise — self-host it, share a link — requires auth (safe to expose), reader-grade public pages, and an upgrade story for instances that hold real data. End-state narrative: "我在自己的 NAS 上跑着 SHCKB，登录写笔记，发布后把链接发给朋友，他们秒开。"

## Ratified Decisions

### M2-D1 — Scope list (dependency order)

1. **DB migration framework + versioned upgrade guards** (foundation; lands before any schema change)
2. **Auth minimal slice** — username-password login + session cookie + first-admin via env (internet-exposed bootstrap mode) + signup OFF; PEP stub becomes real enforcement. Prerequisite: auth library verify spike + new ADR.
3. **Image block** + content-addressed local-fs blob storage (second BlockKindModule)
4. **Publish-time static HTML snapshot** for the public read route (fast/SEO/no-JS readable; no runtime SSR framework)
5. **Deploy & update closure** — production Dockerfile, secrets via env, version surface, documented `docker compose pull && up -d` upgrade flow
6. **Warts** — slug regenerates from title at first publish; copy-link button

### M2-D2 — Image block over code block (owner decision)

Owner: code block's product future is likely *executable* code (remote runners / distributed execution; carryover kernel-adapter/kernel-pyodide are evidence) — a display-only code block now would set product expectations prematurely. Image is also the higher-frequency real note content. **Entire code-block family deferred** until the execution model is designed.

### M2-D3 — Blob model: content-addressed, immutable, local-fs

Blob id = content hash; files under `data/blobs/`; never mutated (new upload = new id). This also resolves the two-state wrinkle: published snapshots reference blob ids, and immutability guarantees old published images survive author-side replacement. **Cut**: thumbnails/srcset, GC/retention (surfaced debt → future backup/retention ADR), S3 adapter (seam only). Alt text required (a11y baseline).

### M2-D4 — Migrations + update are one story, and land first (owner addition)

Owner added 更新方案 + 数据库升级迁移方案 to the scope; rationale ratified by both sides: MVP-2 itself is the first schema change (auth/blob tables), and post-MVP-2 real instances hold real data — every later upgrade without migrations gambles data. Decision details in [ADR-0020](../../decisions/ADR-0020-db-migrations-upgrade.md). Update mechanism for Docker self-host = pull new image tag + restart; migrations auto-apply at startup; version visible at `/api/health`. NOT in scope: auto-update, in-app update UI, single-binary self-update (setup-time PRD M3/M4).

### M2-D5 — Doc debts riding along

- Auth library selection ADR (verify spike first) — the AUDIT debt from authentication PRD.
- Blob storage ADR when step 3 lands.

## Build Log

Phase 1 — migration framework (2026-06-11):

1. **drizzle-kit generate + 自写 applier** 落地 per ADR-0020：journal 表 `skb_migrations`、baseline stamping、降级/篡改双护栏、事务级原子应用。6 个 applier 测试 + api 测试全绿（17/17）。
2. **`idx_blocks_notepage` 索引补进 drizzle schema** 后重新生成 0000——保证 fresh-via-migration 与 baseline 库 schema 完全一致。
3. **`/api/health` 现暴露 `version` + `schemaVersion`**；server 版本号 0.2.0-dev。
4. **真实库 baseline 实测**：owner dev.db（含 newtest 页）重启后 baseline stamp 成功、数据完好、schemaVersion=0。
5. **WAL 备份教训（实战踩到）**：仅拷 `.db` 文件的备份不完整——运行中实例的大部分数据可能在 `-wal` 文件里。升级文档必须写：停容器后拷整个 `data/` 目录（含 `-wal`/`-shm`），或先执行 `PRAGMA wal_checkpoint(TRUNCATE)`。归 Phase 5 文档。

Phase 2 — auth slice (2026-06-11)，详 [ADR-0021]：

6. **better-auth verify 通过**：26 server 测试（bun:test）+ 浏览器登录闭环（401→/login→sign-in→列表→sign-out）。挂载实例 signup 永禁；first-admin 走临时 bootstrap 实例；schema 经 `@better-auth/cli generate` 并入 migration 流（0001_auth.sql；CLI 跑在 Node 下，config 需用 stub db，bun:sqlite 无法在 Node 加载）。
7. **`SHCKB_BASE_URL` 在任何反代后必设**：dev 的 vite proxy 触发 better-auth `INVALID_ORIGIN` 403，设 SHCKB_BASE_URL=前端 origin 解决。这是 operator 文档要点（反代部署同理），归 Phase 5。
8. **ownerId 列推迟**（与 ADR-0021 授权模型一致）：单用户实例下 ownership 列零行为差异；多用户支持时再加列 + backfill。
9. **Windows 进程教训**：后台任务 stop 杀不死孙进程——残留的旧 vite（代理指向 :3000 的无关服务）造成过一轮 200/401 假象排查。清场要按端口找 OwningProcess 杀。
10. **真库 0001 实测**：owner dev.db 应用 auth migration（schema 0→1）+ bootstrap first admin，notepage 数据完好。

Phase 3 — image block + blob storage (2026-06-11)，详 [ADR-0022]：

11. **Content-addressed blob store 落地**：sha256 id / temp+rename 写入 / mime allowlist（无 SVG）/ 10 MiB 上限 / 公开读带 `immutable` 缓存。Migration 0002（schema 1→2 真库实测通过）。30 server 测试全绿（含编码路径穿越拒绝；原始 `../` 在 URL 规范化层就进不了路由）。
12. **Image 是第二个 BlockKindModule**：零特殊化接入 registry / 编辑器 / 公开读——seam 泛化得证（MVP-2 选 image 的架构目的达成）。浏览器 E2E：上传 → 预览 → publish → 匿名公开页 img + alt 全通。
13. **Capability-URL trade-off 已记录**（ADR-0022 §5）；**blob GC 是显式 debt**；备份对象自此 = `data/` 整目录（DB + blobs）。

Phase 4 — publish-time static HTML (2026-06-11)：

14. **快照即静态页**：publish 时用 unified 管线把 publishedDoc 渲染成完整 HTML 存 `published_html` 列（migration 0003），`/notes/:slug` 由 server 直出——快、可索引、无 JS 可读，零运行时 SSR 机制。404 对 missing/private/unpublished 逐字节一致（no-leak）。
15. **Server 渲染器与 SPA RenderView 是平行实现**（不复用 React 组件——依赖方向不对）。已知 drift 风险；第三个渲染消费者出现时抽 shared block-render package。raw HTML 同样 drop（与 web sanitize-by-omission 口径一致），title/alt 等全部转义（测试覆盖 XSS 注入用例）。
16. **Dev 形态**：vite SPA 的 ReadPage 仍走 JSON API（dev 便利）；静态页是 :3210/notes/:slug 与生产形态的 canonical 路径。34 server 测试全绿；真库 re-publish 实测 schema 3 + 2KB 静态页直出。
