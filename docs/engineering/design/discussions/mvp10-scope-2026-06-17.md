### M10-D4 · 搜索派生索引设计（被 pivot displaced）（2026-06-17）

> 注：此节是搜索路线的设计成果，**在 M10-D5 pivot 后整体顺延 MVP-11**。保留是因为这些技术结论
> （派生索引套路、index 维护三选一、`index?()` 契约缝）在 MVP-11 仍然成立，且 chronology 有价值。

我方核证存储形态后确认 backlinks 的原语已就位：页链存 `pagelink` mark、**只存 pageId**
（[schema.ts]），渲 `/p/:id`、改名不坏；[richtext.ts] 的 `linkedPageIds(content)` 注释已写
"future backlink feed"，`extractText(content)` 也在——**richtext 的 searchableText 与 outboundLinks
两个抽取器已存在**。关键 nuance：正向边活在 block 的 content JSON 里（pagelink mark），**不是可查询
的边表**。故 MVP-10 搜索的脊梁 = 一层**派生索引（derived index）**，与 `publishedHtml = f(doc)`、
blob-refs 枚举同一"派生投影"套路：block 写工作态时，每 kind 贡献 `searchableText` + `pageRefs`，
host 落两张派生表（① FTS5 虚表 ② `page_links` 边表），均可从源重建。

index 维护 3 选 1，明确赢家 = **(a) app 层在 block 写入时抽取**（host 调每 kind 的贡献器）；
(b) FTS5 external-content + 触发器**不行**（content 是 kind-owned JSON，SQL 触发器跑不了 kind 抽取器）；
(c) 索引发布快照 HTML**不行**（搜不到草稿，对作者 dogfooding 最差）。(b)(c) 的失败正好反证
**per-kind 抽取必须在应用层**——顺势给 BlockKind 契约加 `index?(content) => {text, pageRefs?}`
贡献点（又一个面的 plugin host API 压测）。HOW 拍板（按 [[feedback-what-vs-how-prd-boundary]] 归我）：
provider = **FTS5 具体实现 + 极薄 SearchProvider 缝**（反-lock，一个实现，不预建 Meili/tsvector）；
[ADR-0008] 是 deprecated legacy，这轮 PRD-informed 重导新 ADR。

### M10-D5 · PIVOT：MVP-10 重定为「视图模式统一」（2026-06-17）

设计正要收口、我方追问"搜索覆盖谁"时，owner 中断并 reframe（顺嘴抛进的一条，却踩中 keystone）：

> "顺便提一嘴。既然做 backlinks 的话，我们文本内的 links 跳转也需要顺便优化一下了。比如说目前
> 存在的页面间跳转，目前的跳转其实是有问题的，你可以看一下，他的逻辑上有什么问题，比如说在
> editor 模式下跳转会直接跳转到 view 页面。**本质上我认为其实是整个项目两种 view 模式的不统一。**"

owner 明确要**根因**而非补丁。我方走 systematic-debugging skill，Phase-1 坐实机制（点链接 → 服务器
→ 302 → ReadPage）：

- pagelink 在编辑器里渲成**裸原生 `<a href="/p/:id">`**（[RichtextRenderView.tsx]），apps/web 里
  **无任何点击拦截**（grep `data-skb-page|navigate(` 在 web 零命中）。
- `/p/:id` 是**服务器路由、不在 SPA router**（[index.ts] 把 `/p/` 交服务器；[main.tsx] 无此路由）→
  点链接 = **整页跳服务器**。
- 服务器（[notepages.ts]）：查不到 / 非 public / 未发布 → **404**；否则 `redirect('/notes/:slug', 302)`
  → SPA 冷启 → **ReadPage（只渲发布快照、standalone）**。

**比 owner 说的更狠的三条后果**：① 每次点链接 = 整页重载（连 reader→reader 也吃全量冷启），是
SPA 完整性的洞；② 草稿 / 私有目标 = 404，**作者无法在自己的链接图里走动**；③ 永远落在发布快照
（看到的是上次发布版，不是当前工作态草稿）。

**根因收紧（用项目自己的词）**：这是 **layer-error**，不是泛泛的"两个模式"。`/p/:id` 原语本身没错
（为公开分享设计：改名不坏、no-leak 404、可贴站外），错在**编辑器把这条"公开分享永链"当成了正文内
导航原语**——一个为"读者 / 对外"作用域设计的东西，被用进了"作者 / 编辑"作用域，正是
[[feedback-framing-anti-patterns]] 那条 layer-error。底下压着的是三条正交轴（mode / state / frame），
而非"两个模式"；且 [ReadPage] 注释明写 "never working state"——**作者没有任何办法像读者一样预览
自己的草稿**，这是不统一的本体。

我方判断：**这是 MVP-10 的拱心石，不是支线**——正文链接跳转、backlink 跳转、搜索结果跳转是**同一个
原语** `navigateToPage(pageId, 当前模式)`，三处全走它。修它不是从 search 岔开，而是 search +
backlinks 脚下的共同地基。给 owner 三档深度（A 只修导航 / A+草稿预览 / 上 B 深度统一）。owner 裁定：

> "我同意上 B 深度统一，然后还有一点是我觉得不能提前就认为在编辑的时候就认为他是发布页的跳转，
> 而在编辑界面也应该可以用，比如说在编辑界面 A 跳转到编辑界面 B，而在发布界面就是在发布界面 A
> 跳转到发布界面 B。这还存在一个，有 chrome 的发布界面和没有 chrome 的发布界面的问题。同时，
> 还存在一个体验优化的问题，比如说浏览位置暂存，回退到上次浏览位置等等。"

**MVP-10 当场重定为「视图模式统一」**，search + backlinks 顺延 MVP-11、落在统一导航原语之上变成
trivial 消费者（搜索结果跳转 / backlink 跳转 = `navigateToPage` 的两个调用点）。owner 同时加进三条
新维度：**mode 整组继承**（edit-A→edit-B、publish-A→publish-B，不能在编辑时就当发布页跳转）、
**有 chrome / 无 chrome 的发布界面**之分、**浏览位置暂存 + 回退**的体验层。

### M10-D6 · 三 surface 模型（2026-06-17）

我方把 owner 三条新维度收进**三正交轴**模型（mode = edit|read / state = working|published /
frame = in-app|bare），8 格大多无意义，收敛成**真实 3 个 surface**：

| Surface | mode | state | frame | 标识 |
|---|---|---|---|---|
| **Editor** | edit | working | in-app | **id** |
| **In-app View** | read | working ⇄ published（可切） | in-app | **id** |
| **Public**（分享件） | read | published | bare | **slug** |

关键收敛（owner 拍了两点）：① **Editor 与 In-app View 收敛成同一页（by id）的两个 mode、原地
toggle**——对应历史那条"一个 URL 跨 edit/view"不变量，只是现在对 state 诚实（In-app View 自带
working⇄published 档，补上"像读者一样看我的草稿"缺口）；② owner 要的**"有 chrome 的发布页"并进
In-app View 的 published 档**（而非单列第四面），**"没 chrome 的发布页"= Public**（chrome 与否 =
工作区内 vs 看一个独立分享件）。三机制：mode/state 整组继承（`navigateToPage(id)` 从当前 surface
读 (mode,state) 解析到同一 surface）/ 标识分法（app surface 一律 by id 永不对草稿 404，Public by
slug 可读·SEO）/ 位置层（每 `(pageId, surface, state)` 暂存 scroll + 激活块，`navigateToPage` 预留
stash/restore hook）。

PRD/ADR 账：这动了已漂移的"一个 URL 跨 edit/view" notepage 不变量 + 两态核心，要落一条新
PRD-informed ADR（路由 / 标识 / 模式模型）+ 更新 notepage-view/editing PRD；按
[[feedback-prd-master-adr-downstream]] PRD 先行、ADR 下游。

### M10-D7 · 流畅性 + block 粒度链接（2026-06-17）

owner 认了三 surface 模型，再追两点：

> "我觉得你的方案是合理的。我在考虑另一件事情，这个方案，会不会影响流畅性。比如说我们在编辑器面里
> 跳转，会不会出现重新加载界面的情况，而不是只加载所需的那个 notepage 这种重加载情况？以及还有
> 一个问题就是，这个链接还可能与另一个功能有关，或者本质上是同一种能力。即一个链接指向同一页面的
> 某一个 block，一个链接指向其他页面的某一个 block，这种细节到这种程度的链接能否实现？"

我方核证两段代码，答案都是"能，而且地基已在"：

- **流畅性 = 不会重加载，而且这正是修复的本体**：[Shell.tsx] 是 layout route，渲 `<Outlet/>`，
  sidebar / 目录树 / 主题 / overlay / auth 全挂 Shell，React Router 只换 Outlet → 客户端
  `navigate('/edit/:otherId')` = chrome 整套留挂、只拉那一页、**零浏览器重载**。当前 bug 的根
  （`<a href="/p/:id">` 原生跳服务器 = 整 app 拆了重建）反衬：把它换成客户端 `navigateToPage`，
  **消除重载就是这次修复的定义，不是附带好处**。诚实 nuance：[EditorPage] 的 `<Editor key={pageId}>`
  会让内容盘按页 remount（故意的干净起点，廉价子树重挂、非 app 重载）；想更顺可留旧视图防闪 /
  hover 预取，但重 chrome 永不重挂。→ **不变量①：页间导航一律 client-side，chrome 常挂，只拉目标页
  payload。**
- **block 粒度链接 = 能，同一个原语**：DOM 里每块已 `data-block-id`（[GridCanvas.tsx]，块 id 是
  `(notepage_id, id)` 复合主键、稳定）→ scroll-to-block = `querySelector` + `scrollIntoView` + 瞬时
  高亮，锚点现成。链接目标统一成 `{pageId, blockId?}`，一个原语 `navigateToPage(pageId, blockId?)`
  吃三种：blockId 缺省→页链；pageId==当前页+blockId→**纯滚动高亮、根本不导航**（同页块链）；
  pageId!=当前+blockId→同 surface 客户端跳页 + 落地后滚到块（跨页块链）。继续往上合并 backlink /
  搜索结果也能指到块。→ **不变量②：链接目标 = (page, block?)，贯穿 pagelink mark / backlink /
  搜索结果。**（两态 nuance：块锚在当前 surface 的 state 里解析，published 里发布后才加的块优雅降级
  落到页、不滚；published canvas 要补同样 `data-block-id`，trivial。）

分期建议（我方提，owner 后续 greenlight）：MVP-10 把**原语 + pagelink mark 做成 block-capable**
（同页/跨页块跳在原语层就通）+ authoring affordance 给最小口（"右键块 → 复制块链"，复用画布右键菜单）；
**block-targeted 的消费者（搜索跳块 / backlink 跳块）随 MVP-11**。

### M10-D8 · link 升一等、跨 kind 能力 + 落地疑虑（2026-06-17）

owner 抛出一串"落地会真崩"的点：

> "在落地的时候肯定还会遇到许多个问题，我简单列几个。比如不同 block 里的不同形式的链接如何处理？
> 比如说 markdown 是比较简单的，可以用自带的超链接，也可以我们单独加解析器。那对于 richtext block
> 怎么办？对于未来会加的真正的 canvas block 怎么办？……同时我们还需要考虑，导出导入的时候会不会
> 破坏这些链接？如果会，我们该怎么办？以及未来可能需要支持导出 pdf？如何处理这链接？未来的 agent
> 又如何根据链接访问他需要的内容？用户添加链接是不是需要一个通用的方式，比如说使用 MVP11 实现的
> 搜索引擎实现选择链接内容？等等"

我方先判**事实问题**（导出/导入会不会破坏链接），因为它定地基：exporter 把 page/block/folder 的 id
**原样序列化**（[exporter.ts]）；importer 是**空实例全量恢复 + id 原样插入**（[importer.ts]，gate 1
拒非空实例，无 remap、无重新生成），content 是 opaque JSON 整存——所以 `pageId#blockId` 引用导入后
指向同一 id，**今天链接完好**。⚠️ 但完整性只靠"空实例 + 保 id"这条不变量撑着；**trigger = 未来的
选择性 / 合并导入**（memory 已记为 deferred）：id 会撞 → 要 remap → 内部链接必须跟着重写。

元框架（我方定，owner greenlight）：**不靠"全实现"回答，靠"建对缝 + 分期落"**——把链接升成**一等、
跨 kind 能力**，目标 `LinkRef = {pageId, blockId?}`，三条缝：

| 缝 | 方向 | 谁实现 | 何时 |
|---|---|---|---|
| **① 抽取** `links(content): LinkRef[]` | kind → host | 每 kind 按自己存法吐统一 LinkRef | **MVP-10** |
| **② 导航** `navigateToPage(ref)` | host → 渲染的链接 | host 唯一 client-side 原语 | **MVP-10** |
| **③ 创作** `pickLinkTarget(): Promise<LinkRef>` | kind → host | 搜索驱动的通用选择器 | **MVP-11** |

逐条回 owner：

- **不同 kind 不同形式**：缝 ① 兜住。markdown = 超链接 href 指 `/p/:id(#block)`（**零新语法**，复用
  永链；wikilink 解析器是可选增强）；richtext = pagelink mark（加 blockId）；未来 canvas block = 链接
  挂形状上、照样吐 `LinkRef[]`。**kind 拥有表达，host 拥有 LinkRef 类型 + 导航 + 创作。** 缝 ② 实现
  形状 = 画布层一个**委托点击处理器**，认 `data-skb-page`/`/p/:id` 标记 → preventDefault →
  `navigateToPage`，任何 kind 渲染带标记的 anchor 就自动接上、不必各自接线。
- **导出/导入**：今天安全；未来 selective import 的 remap 因为缝 ① 是统一抽取，是 host 层一次
  `extract→remap(idMap)→rewrite`（给缝 ① 配对偶 `remapLinks(content, idMap)`），不是逐 kind 拆 JSON。
  **MVP-10 不建 remap，但 ADR 写明这条债 + trigger。**
- **PDF 导出**：defer。LinkRef 与渲染目标无关——MVP-10 零工作，只要别把链接写死成 HTML href。
- **agent 按链接访问**：defer 到 MVP-11。`resolveLink(ref)→content` 对 host 是 thin layer；而且
  **backlinks 的 `page_links` 边表 = agent 的链接图索引**，缝 ① 抽出的 LinkRef 就是 agent 的出边——
  **一份 link graph，人走也机器走。**
- **通用加链接方式**：就是缝 ③。通用 `pickLinkTarget()` 由 MVP-11 搜索驱动，所有 kind 共用；MVP-10
  先把这个 HostServices 签名占好（kind 对着它编程），实现随 MVP-11 落地。

**串起来**：MVP-10 铺 `LinkRef + 抽取 + 导航`；MVP-11 搜索点亮 `创作 + block-targeted 搜索/backlink
+ agent 遍历`；再未来 canvas 链接（缝①）、PDF（渲染目标）、selective-import remap（完整性债）各自
插槽——**这反过来确证了 M10-D2 的排序是对的。**

## 设计桩（scope 钉定，详见待写 spec / 待立 ADR）

> 本轮**只到 scope 定稿 + 设计成形 + greenlight 写 spec**；实现尚未开始，下列为讨论中钉定的设计骨架。

**三 surface 模型**：Editor（edit / working / in-app / by id）⇄ In-app View（read / working⇄published
可切 / in-app / by id）/ Public（read / published / bare / by slug）。Editor 与 In-app View 是**同一页
by id 的两个 mode，原地 toggle**；"有 chrome 的发布页"并进 In-app View 的 published 档，"无 chrome 的
发布页"= Public。

**两条导航不变量**：① 页间导航一律 **client-side、surgical**——chrome（Shell layout route）常挂，只换
Outlet、只拉目标页 payload，零浏览器重载；② 链接目标 = **`{pageId, blockId?}`**，一个原语
`navigateToPage(pageId, blockId?)` 吃 页链 / 同页块跳（纯滚动不导航）/ 跨页块跳，贯穿 pagelink mark /
backlink / 搜索结果。

**Link = 一等、跨 kind 能力，三条缝**：① 抽取 `links(content): LinkRef[]`（kind→host，MVP-10）；
② 导航 `navigateToPage(ref)`（host 唯一 client-side 原语 + 画布层委托点击处理器认 `/p/:id`/
`data-skb-page` 标记，MVP-10）；③ 创作 `pickLinkTarget(): Promise<LinkRef>`（搜索驱动通用选择器，
**签名 MVP-10 占好、实现 MVP-11**）。kind 拥有表达，host 拥有 LinkRef 类型 + 导航 + 创作。

**标识分法**：app surface 一律 **by id**（稳定、永不对草稿 404）；Public **by slug**（可读·SEO，改名走
`/p/:id`→slug 桥，对外未发布 = 正当 404）。`/p/:id` 永链原语保留其本职（公开分享：改名不坏、no-leak、
可贴站外），不再被当成正文内导航原语。

**位置层**：每 `(pageId, surface, state)` 暂存 scroll + 激活块，history 后退 / 重入还原；
`navigateToPage` 预留 stash/restore hook（别事后补）。

**分期边界**：MVP-10 = 视图模式统一 + LinkRef 抽取/导航 + pagelink mark 做成 block-capable + 最小
authoring（右键块→复制块链）；MVP-11 搜索 = 创作选择器 + block-targeted 搜索/backlink + agent 遍历 +
语义/related。

### 续盘设计桩（覆盖 D6 三-surface；详见订正 spec / ADR-0031 订正）

> 本日续盘**取代** D6/旧设计桩里的「三 surface 模型 + slug 标识」。下列为 2026-06-18 收敛的新骨架。

**四 surface / 两轴（audience × scope）**：edit `/edit/:id`、view `/view/:id` = **作者·live(working)·
in-Shell**（一体两面，by id）；read `/read/:id` = **公开·published·in-Shell·整库**外链（Shell+公开目录
侧栏可浏览）；note `/notes/:id` = **公开·published·bare·单页**外链。chrome 之有无**由 scope 推出**（逛库
要 chrome / 单页不要）。Shell 已 auth-aware（作者 `getTree` vs 访客 `getPublicTree`），故主要是
**重命名/重分类既有路由**。

**URL 标准 = 全 id（SSOT）**：所有 surface URL 统一 by **id**；slug 退出 URL；slug-as-alias（id canonical、
slug 302→id）**留作未来**。收益：id 比 slug 更 rename-safe / 解析器塌成「同 surface + 同 id」/ 退役发布期
slug 物化（4 写点 + `publicIdToSlug`）。导出/导入链接完整性 = 独立的 import-时 id-remap 关切，不靠全-id 解决。

**Q1 修复**：`/read/` 升为 in-Shell app surface，点链接落 `/read/:id`、chrome 守恒、客户端；`/notes/` 单页
内点链接留 `/notes/`。frame 轴（owner 开盘即点）**结构上守住**。

**Q2 现状 + 修复**：页内块链今天靠手写 `[label](/p/:id#blockId)` 工作；**copy-link 摩擦修掉**（「Copy link
to block」改复制路径形 `/p/:id#blockId`，或令解析器也认 app-origin 完整 URL）；点选 picker（含 richtext
块级目标）= ③ 创作缝，**仍留 MVP-11**。

### M10-D17 · 收尾优化①：preview 作为浏览模式（2026-06-18）

回改全绿、正报 finish 选项时，owner 要先做一轮优化：

> "收尾前做一轮优化吧。比如这个 preview，我觉得可以做到左侧 chrome 而不是顶部，改成切换功能，即改成
> preview 模式，则点击所有的 note 都是 view 模式。"

我方落地前先盘出**真根因**：编辑器顶栏 `preview ◉` 只是单页 link，而 **sidebar 作者页行硬编码 `/edit/`**
（[Sidebar.tsx]）——不参与 D14/D15 立的「守面」不变量。故在 `/view/A` 预览时点 sidebar 的 B 被弹回
`/edit/B`，preview 是单页死角。reframe = 把守面从「正文内链接」延伸到「左侧 chrome」。owner（AskUserQuestion）
裁 **Full — now**。落地（spec §12）：sidebar 作者页行**跟随当前作者面**（edit/view）；左 chrome 一个
**edit⇄preview 切换 = `surfaceOf(pathname)` 的呈现**（从 URL 调和、无隐藏 mode、不落 localStorage）；编辑器
顶 `preview ◉` + view 顶 `edit ✎` 两个 per-page flip **折叠**进这一个切换。

**全套 e2e 首次全跑逮出 review 盲点**：之前 final review **只跑了 page-links e2e**，本轮第一次跑**全套**就红
3 条——1 条是新切换的 `getByText('edit')` 与 autofit 测的右键菜单项**文案撞**（改 menuitem role 定位），
**另 2 条是全-id 的遗漏债**：`autofit-fix`/`autofit-publish-clip` 仍 `goto('/notes/:slug')`，全-id 后 `:id` 路由
下 slug≠id → 发布页 not-found（D15 的 slug→id 扫荡只跑 page-links 时漏了这两个）。三条全修、全套 e2e 转绿。
**教训（承 D11）再证**：单测全绿 + 只跑一个 e2e ≠ 安全；收尾必须全套真浏览器跑。

### M10-D18 · 收尾优化② + 终裁 §11.1：登录态「以访客预览」+ 底部账户区（2026-06-18）

owner 实跑时问：

> "一个登录了的网页怎么才能看 read 界面呢？"

我方坐实缺口：`/read/:id` 登录态可直敲但①只对已发布页有效、②sidebar 还是作者目录、点了被弹回 edit/view
→ read 登录态不可浏览；这正是 D14 反压、D6 续盘 Open Item 里挂着的「公开面对作者是否走公开投影」。owner 裁
**这轮补**（不留 MVP-11——那轮要扛搜索/AI 的大头）：

> "还是要补的，不过 MVP11 接下来可能要负责更大的任务…所以在这一轮补吧"

形状（AskUserQuestion）= **单独「以访客预览」入口**（非三态切换）+ **sign in/out 移底部悬浮账户区（③）这轮一起做**。
**§11.1 就此终裁**：登录作者进公开面（read/note）看**访客真正的公开目录**（public 投影），非作者全鞋柜。落地
（spec §13 / PRD pass 7 / ADR-0031 §11.1 注）：sidebar 目录改**按 surface 取**（`chromeAudience`：read/note→public
投影即使登录）；底部 sticky **账户区**承载 identity + sign out + 「以访客预览」（作者面）/「退出预览→/edit」
（预览中）+ admin settings；预览中**隐藏 edit/preview 切换**。「以访客预览」目标 = 当前页若已发布否则首篇
已发布否则禁用（`viewAsVisitorTarget`）。

### M10-D19 · ②（记录，不这轮做）：发布状态标识多态（2026-06-18）
